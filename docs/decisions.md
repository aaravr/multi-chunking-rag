2026-03-13: Context: Agents were tightly coupled via an in-process synchronous MessageBus, preventing independent deployment and horizontal scaling. Decision: Introduce Kafka-backed Agent-to-Agent (A2A) communication layer as an alternative to the in-process bus. Components: (1) KafkaEnvelope + serde layer (agents/serde.py) wraps AgentMessage with correlation_id and reply_topic for request-response over async topics. (2) KafkaBus (agents/kafka_bus.py) implements the same send() interface as MessageBus — produces to agent.{name}.requests, blocks on reply topic polling with correlation matching, preserving the orchestrator's synchronous ReAct loop. (3) AgentRunner (agents/agent_runner.py) runs any agent as an independent Kafka consumer process: `python -m agents.agent_runner --agent retriever`. (4) Bus factory (agents/bus_factory.py) selects KafkaBus or MessageBus based on ENABLE_KAFKA_BUS config with graceful fallback. ENABLE_KAFKA_BUS defaults to false; existing in-process behaviour unchanged. Consequences: Each agent can be deployed, scaled, and upgraded independently; Kafka provides durable message log for replay and debugging; correlation-based request-response preserves synchronous ReAct semantics; no changes needed to agent handler code. Alternatives considered: gRPC point-to-point (rejected — no event log, tight coupling); Redis Streams (rejected — less mature ecosystem for this pattern, Redis already used for working memory); HTTP REST (rejected — no backpressure, no replay).
2026-03-13: Context: The initial Kafka A2A layer lacked enterprise resilience, observability, and testability. Each agent was a function wired to Kafka — not an independently deployable, independently scalable component. Decision: Harden Kafka A2A to enterprise grade with hexagonal architecture. Changes: (1) Schema versioning — envelope_version field with backward-compatible deserialisation; SchemaVersionError for unsupported versions; enables safe rolling deployments. (2) Distributed tracing — TraceContext (trace_id, span_id, parent_span_id) propagated in every envelope; child spans created at each hop. (3) Error taxonomy — ErrorCode enum (NONE, HANDLER_CRASH, TIMEOUT, SCHEMA_MISMATCH, DLQ_ROUTED, CIRCUIT_OPEN, IDEMPOTENT_DUPLICATE, VALIDATION_ERROR, UNKNOWN) for operational triage. (4) Circuit breaker — per-agent CLOSED→OPEN→HALF_OPEN state machine; configurable failure_threshold and cooldown_s; prevents cascading failures. (5) Retry policy — exponential backoff with jitter; distinguishes transient vs permanent errors; max_retries, base_delay, max_delay all configurable. (6) Idempotency store — in-memory LRU keyed by correlation_id with TTL; prevents duplicate processing under at-least-once delivery. (7) Dead Letter Queue — poison pills and exhausted-retry messages routed to agent.{name}.requests.dlq; preserves original envelope for replay. (8) Health checks — alive/ready/heartbeat tracking for K8s/ECS liveness/readiness probes. (9) Metrics collector — thread-safe counters (sent, received, errors, timeouts, retries, DLQ depth, orphans), latency histograms (p50-p99), gauges (circuit state, consumer lag, inflight). (10) Hexagonal AgentCore — pure business logic with InboundPort, OutboundPort, MetricsPort; KafkaOutboundAdapter for production, StubOutboundAdapter for unit tests. (11) Manual offset commit — at-least-once delivery guarantee in AgentRunner. (12) Compression — LZ4 default, configurable. (13) Security — SASL/SCRAM + TLS config propagated to all producers/consumers. (14) Async fan-out — send_async() returns Future for parallel agent delegation. (15) Event-driven reply resolution — threading.Event replaces busy-wait polling. (16) Graceful shutdown with in-flight drain. Test pyramid: 80 tests (unit: serde/resilience/metrics/AgentCore with stubs; integration: runner with mock transport; contract: topic naming/error codes/envelope invariants). Consequences: Each agent is independently deployable, independently scalable, fully testable without Kafka; zero regressions on existing 60 tests. Alternatives considered: OpenTelemetry SDK (rejected — adds heavy dependency for trace propagation, built-in envelope tracing sufficient for now); external idempotency (Redis/Postgres) — designed for but deferred, in-memory LRU sufficient for single-instance; Avro/Protobuf schema registry — designed for via envelope_version but deferred, JSON with version field provides adequate evolution path.
2026-03-13: Context: Working memory (§6.1) for the orchestrator ReAct loop was stored as ephemeral Python instance variables, preventing multi-worker state sharing, streaming across requests, and atomic budget tracking. Decision: Introduce WorkingMemoryStore abstraction (agents/working_memory.py) with two backends — DictBackend (in-process, zero dependencies) and RedisBackend (multi-worker, TTL-managed auto-expiry). Redis is enabled by default via ENABLE_REDIS_WORKING_MEMORY=true with graceful fallback to DictBackend when Redis is unreachable. Key schema: wm:{query_id}:{state|plan|evidence|trace|budget} with 15-minute TTL. Budget uses Redis DECRBY for atomic token accounting. Evidence and trace use RPUSH for append-only semantics. Factory singleton pattern matches existing model_registry.py pattern. Consequences: Orchestrator state survives across requests; multi-worker deployments can share ReAct loop state; abandoned queries auto-expire; existing single-process deployments work unchanged via fallback. Alternatives considered: Postgres-backed working memory — rejected due to unnecessary write amplification for ephemeral state; pure in-process only — rejected as it blocks Phase 8 multi-worker scaling.
